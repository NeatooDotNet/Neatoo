# Neatoo 0.20.1

**Release Date:** 2026-03-13
**Type:** Bug Fix
**Breaking Changes:** No

---

## Summary

Fixed a crash where `WaitForTasks()` fails when an `AddActionAsync` handler awaits a `LazyLoad<T>` whose child is fetched via a `[Remote]` factory method. The root cause was the JSON converter replacing constructor-created `LazyLoad<T>` instances during deserialization, stripping the loader delegate.

---

## What's New

### LazyLoad Deserialization Fix

`NeatooBaseJsonTypeConverter` now **merges** deserialized state (`Value`, `IsLoaded`) into existing `LazyLoad<T>` instances via a new internal `ILazyLoadDeserializable` interface, instead of replacing them. This preserves the loader delegate set in the constructor.

**Before (broken):** Converter overwrites the constructor-created `LazyLoad` with a deserialized instance that has no loader. Awaiting the `LazyLoad` on the client throws `InvalidOperationException`.

**After (fixed):** Converter finds the existing instance and calls `ApplyDeserializedState` to merge server-side state. The loader delegate survives the round-trip.

### Correct LazyLoad Pattern

`LazyLoad<T>` must be created in the **constructor**, not in `[Fetch]` or `[Create]`. The constructor runs on both server and client during DI-based deserialization, ensuring the loader delegate is always present.

```csharp
public MyEntity(..., IChildFactory childFactory, ILazyLoadFactory lazyLoadFactory) : base(services)
{
    LazyChild = lazyLoadFactory.Create<IChild>(async () =>
    {
        return await childFactory.Fetch(this.Id); // this.Id resolved at load-time
    });
}
```
