# Neatoo 0.16.0

**Release Date:** 2026-03-06
**Type:** Bug Fix
**Breaking Changes:** No

---

## Summary

Fixes `LazyLoad<T>` properties being silently dropped during client-server serialization. Entities with `LazyLoad<T>` properties now correctly preserve their values when transferred between client and server via `NeatooBaseJsonTypeConverter<T>`.

---

## Bug Fixes

### LazyLoad Properties Not Serialized in Client-Server Scenarios

`LazyLoad<T>` properties on Neatoo domain objects were silently dropped during serialization through `NeatooBaseJsonTypeConverter<T>`. Any entity with a `LazyLoad<T>` property (e.g., `public LazyLoad<OrderLineList> OrderLines { get; private set; }`) lost that data when transferred between client and server.

**Root cause:** The custom JSON converter only handled two property categories:
1. PropertyManager entries (partial properties using `Getter<T>()`/`Setter()`)
2. `IEntityMetaProperties` (`IsModified`, `IsNew`, etc.)

`LazyLoad<T>` properties are regular C# properties that fell outside both categories.

**Fix:** Added `LazyLoad<T>` property detection via reflection on the concrete entity type to both the Write and Read paths of `NeatooBaseJsonTypeConverter<T>`. The detection is independent of the existing `editProperties` mechanism, so it works for both `EntityBase` and `ValidateBase` entities.

**Key design details:**
- Delegates to `JsonSerializer.Serialize`/`Deserialize` for the `LazyLoad<T>` wrapper, passing the same `JsonSerializerOptions` so nested `IValidateBase` entities inside `LazyLoad<T>.Value` go through the Neatoo converter for `$id`/`$ref` handling
- Pre-loaded `LazyLoad<T>` (`IsLoaded=true`, `Value` set) survives round-trip
- Unloaded `LazyLoad<T>` (`IsLoaded=false`, `Value=null`) survives round-trip
- Null `LazyLoad<T>` properties (not yet assigned) are skipped
- Post-deserialization, `LoadAsync()` throws `InvalidOperationException` (loader delegate is not serializable) — this is existing, documented behavior

---

## Migration

No code changes required. This is a pure bug fix with no API changes.

`LazyLoad<T>` properties on entities will now automatically survive serialization round-trips. If you had workarounds for missing lazy-loaded data after remote calls, those workarounds can be removed.
