# Neatoo 0.23.0

**Release Date:** 2026-03-15
**Type:** Breaking Change
**Breaking Changes:** Yes -- `LazyLoad<T>.Value` auto-trigger removed; `GetAwaiter()` removed

---

## Summary

Reverts the v0.21.0 auto-trigger behavior on `LazyLoad<T>.Value`. The `.Value` getter is now a passive read that returns the current value (or `null` if not loaded) with no side effects. `LoadAsync()` is the sole mechanism for triggering loads. `GetAwaiter()` is removed -- `await lazyLoad` no longer compiles.

This revert is based on real-world feedback from zTreatment: convenience accessors that read `.Value` (e.g., `hub.Treatment`, `hub.SymptomsAssessment`) inadvertently triggered loading when accessed by the serialization pipeline, computed property evaluation chains, or test assertions. The auto-trigger collapsed two distinct intents ("give me what's loaded" vs "load it if needed") into one accessor.

---

## What Changed

### Value Is a Passive Read

`LazyLoad<T>.Value` no longer triggers loading. It returns the current state: `null` if not loaded, the loaded value otherwise.

**Before (0.21.0-0.22.0):**
```csharp
var child = entity.LazyChild.Value; // Triggered fire-and-forget load
```

**After (0.23.0):**
```csharp
var child = entity.LazyChild.Value; // Returns null -- no side effect
```

Two patterns for consumers:
- **Imperative code** (tests, domain logic, `OnInitializedAsync`): `await entity.LazyChild.LoadAsync()`
- **Binding** (Razor, WPF): `entity.LazyChild.Value` for display, `LoadAsync()` in `OnInitializedAsync()`

### GetAwaiter Removed

`await lazyLoad` syntax no longer compiles. Use `await lazyLoad.LoadAsync()` instead.

```csharp
// Before
var child = await entity.LazyChild;

// After
var child = await entity.LazyChild.LoadAsync();
```

### TriggerLoadAsync Removed

The internal `TriggerLoadAsync()` method (the fire-and-forget wrapper used by the old Value getter) is removed as dead code.

### WaitForTasks Integration Preserved

`ValidateBase.WaitForTasks()` continues to await in-progress LazyLoad children via `PropertyManager.WaitForTasks()`. Explicit `LoadAsync()` calls produce running tasks that WaitForTasks awaits -- this behavior is unchanged.

---

## Breaking Changes

### LazyLoad<T>.Value No Longer Triggers Loading

Code that relied on `.Value` access to start the deferred fetch must now call `LoadAsync()` explicitly.

### GetAwaiter() Removed

`await lazyLoad` no longer compiles. Replace with `await lazyLoad.LoadAsync()`.

### Removed APIs

| Removed | Replacement |
|---------|-------------|
| `LazyLoad<T>.GetAwaiter()` | `await lazyLoad.LoadAsync()` |
| `LazyLoad<T>.TriggerLoadAsync()` (internal) | Call `LoadAsync()` explicitly |
| Auto-trigger in `Value` getter | Call `LoadAsync()` explicitly |

---

## Migration

### Step 1: Replace `await lazyLoad` with `await lazyLoad.LoadAsync()`

Search for `await` followed by a `LazyLoad<T>` expression. The compiler will flag these as errors since `GetAwaiter()` no longer exists.

```csharp
// Before
var phoneList = await this.PersonPhoneList;

// After
var phoneList = await this.PersonPhoneList.LoadAsync();
```

### Step 2: Add explicit LoadAsync() where Value auto-trigger was relied on

If your code accessed `.Value` expecting it to start a load (e.g., in Razor pages), add an explicit `LoadAsync()` call:

```csharp
// Before (Razor page relied on auto-trigger)
@{ var lines = Model.OrderLines.Value; }

// After (call LoadAsync in OnInitializedAsync)
protected override async Task OnInitializedAsync()
{
    await Model.OrderLines.LoadAsync();
}

// Razor template unchanged -- .Value now returns the loaded data
@{ var lines = Model.OrderLines.Value; }
```

### Step 3: AddActionAsync handlers

If you had `AddActionAsync` handlers that used `await parent.LazyChild`, update to `await parent.LazyChild.LoadAsync()`:

```csharp
RuleManager.AddActionAsync(async parent =>
{
    var child = await parent.LazyChild.LoadAsync();
    if (child != null)
    {
        parent.LoadedData = child.Data;
    }
}, p => p.Trigger);
```
