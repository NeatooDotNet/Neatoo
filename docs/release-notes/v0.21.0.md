# Neatoo 0.21.0

**Release Date:** 2026-03-13
**Type:** Feature
**Breaking Changes:** No (behavioral change — see Migration below)

---

## Summary

`LazyLoad<T>.Value` now auto-triggers the deferred fetch as fire-and-forget when accessed, enabling Razor databinding to work without explicit `await` boilerplate. `ValidateBase.WaitForTasks()` now awaits in-progress LazyLoad children.

---

## What's New

### LazyLoad Auto-Trigger on Value Access

Previously, `LazyLoad<T>.Value` returned `null` without triggering the load — requiring explicit `await lazyLoad` or `await lazyLoad.LoadAsync()`. This forced developers to add manual kickoff code in Razor pages.

Now, accessing `.Value` automatically starts the load when:
- The value hasn't been loaded (`!IsLoaded`)
- No load is in progress (`!IsLoading`)
- A loader delegate exists (not a deserialized instance without one)

`IsLoading` and `IsLoaded` remain pure state checks with no side effects.

**Razor databinding pattern:**

```razor
@if (Model.OrderLines.HasLoadError)
{
    <MudAlert Severity="Severity.Error">@Model.OrderLines.LoadError</MudAlert>
}
else if (Model.OrderLines.Value == null)
{
    <LoadingSpinner />
}
else
{
    <OrderLinesList Items="@Model.OrderLines.Value" />
}
```

Accessing `.Value` triggers the load. `PropertyChanged` fires when the load completes, causing Blazor to re-render automatically.

### WaitForTasks Awaits LazyLoad Children

`ValidateBase.WaitForTasks()` and `WaitForTasks(CancellationToken)` now await in-progress LazyLoad children. Previously, only `RunningTasks` and `PropertyManager` tasks were awaited — auto-triggered lazy loads could complete after `WaitForTasks()` returned.

This ensures `await entity.WaitForTasks()` before `Save()` waits for all pending lazy loads to finish.

### Error Handling

- **Fire-and-forget path (Value getter):** Exceptions are caught and surfaced via `HasLoadError` / `LoadError` / `IsValid=false`. No unobserved task exceptions.
- **WaitForTasks path:** Exceptions propagate to the caller for explicit handling.

---

## Migration

### Behavioral Change: Value Access Triggers Load

Code that accessed `.Value` on an unloaded `LazyLoad<T>` expecting `null` with no side effects will now trigger a load. This is the intended behavior — `.Value` means "I want the data."

To check state without triggering: use `IsLoaded` or `IsLoading` (these remain side-effect-free).

**Before:**
```csharp
// No side effect — just returns null
var child = lazyLoad.Value;
```

**After:**
```csharp
// Triggers load if not loaded — returns null synchronously, load runs async
var child = lazyLoad.Value;

// To check without triggering:
if (lazyLoad.IsLoaded) { /* safe — no side effect */ }
```
